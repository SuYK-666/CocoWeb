## 二、风险识别

原 Web 任务流程中，用户创建分析任务后，系统使用 `job_id` 查询运行状态、读取 SSE 实时日志、下载 Markdown/HTML/DOCX 报告。`job_id` 本质上是资源定位标识，不应同时作为访问授权凭据。

如果 `job_id` 出现在浏览器历史、错误日志、截图或他人转发的链接中，第三方可能直接访问以下敏感内容：

- 分析任务实时日志；
- 最终分析报告；
- 报告文件下载地址；
- 仓库结构、依赖、代码洞察、安全风险判断等过程信息。

因此，本次识别出的主要安全风险是：未授权读取任务日志与报告产物。

## 三、安全约束如何进入 AI 交互

我在动手修改前，将风险识别结论转化为结构化约束，并在与 Codex 的交互中持续使用。约束重点如下：

- 每个任务创建时生成独立访问 token；
- `job_id` 只用于定位任务，不能作为唯一访问凭据；
- `/api/result/<job_id>`、`/api/stream/<job_id>`、`/api/download/<job_id>/<fmt>` 三类接口必须统一校验 token；
- 缺失 token 返回 401，错误 token 返回 403；
- token 必须使用安全随机数生成；
- token 比较必须使用常量时间比较；
- 前端可以在当前任务生命周期中保存 token，但不能写入 localStorage；
- 必须新增单元测试覆盖无 token、错 token、正确 token 等情况；
- 禁止修改个人目录以外的任何代码或文档。

完整约束文档同步记录在 `docs/constraint-doc.md`，Prompt 交互记录同步记录在 `docs/prompt-records.md`。

## 五、AI 生成结果中与安全约束相关的关键片段

### 1. 使用安全随机数生成任务访问令牌

```python
access_token = secrets.token_urlsafe(32)
```

安全意义：令牌由 Python 标准库 `secrets` 生成，不依赖时间戳、仓库名、用户名或可预测序列，降低猜测风险。

### 2. 从请求中读取访问令牌

```python
def _get_request_token() -> str:
    auth_header = request.headers.get("Authorization", "")
    if auth_header.lower().startswith("bearer "):
        return auth_header.split(" ", 1)[1].strip()
    return str(request.args.get("token", "")).strip()
```

安全意义：同时支持 `Authorization: Bearer` 与查询参数 `token`。其中查询参数用于浏览器原生 EventSource 和下载链接场景，Bearer 方式为后续 API 调用保留更规范的扩展路径。

### 3. 统一授权校验与常量时间比较

```python
def _authorize_job(job_id: str) -> dict[str, Any]:
    job = _get_job(job_id)
    if not job:
        abort(404, description="job not found")

    expected_token = str(job.get("access_token", ""))
    provided_token = _get_request_token()
    if not expected_token or not provided_token:
        abort(401, description="missing job access token")
    if not secrets.compare_digest(expected_token, provided_token):
        abort(403, description="invalid job access token")
    return job
```

安全意义：将授权判断集中到一个函数中，避免某个敏感接口漏掉检查；`compare_digest` 用于降低普通字符串比较可能带来的时序侧信道风险。

### 4. 三类敏感接口统一调用授权校验

```python
@app.get("/api/stream/<job_id>")
def stream(job_id: str) -> Response:
    job = _authorize_job(job_id)
```

```python
@app.get("/api/result/<job_id>")
def result(job_id: str) -> Response:
    _authorize_job(job_id)
```

```python
@app.get("/api/download/<job_id>/<fmt>")
def download(job_id: str, fmt: str) -> Response:
    _authorize_job(job_id)
```

安全意义：实时日志、结果查询、报告下载都属于任务敏感数据读取，必须具有同一安全边界。

### 5. 前端自动携带 token，但不持久化任务 token

```javascript
const state = {
    currentStep: 0,
    eventSource: null,
    resultUrl: "",
    accessToken: "",
    jobId: "",
    logCursor: 0,
    pollTimer: null,
    finalResolved: false,
    streamWarned: false,
};
```

```javascript
response = await fetch(withQuery(state.resultUrl, { t: Date.now(), token: state.accessToken }), {
    cache: "no-store",
    headers: { "Cache-Control": "no-cache" },
});
```

安全意义：任务访问 token 只保存在本次页面运行状态中，没有写入 localStorage；轮询时自动携带 token，保证新增授权逻辑不会破坏正常功能。

## 六、发现偏差或问题时的交互记录

### 偏差 1：下载链接需要同步携带 token

发现情况：后端给下载接口增加 `_authorize_job` 后，如果最终返回给前端的 `downloadUrls` 不带 token，用户点击下载会被新增授权逻辑拦截。

修正动作：在 `_run_analysis_job` 构造 `downloadUrls` 时追加当前任务的 `job_access_token`。

```python
downloadUrls = {
    fmt: f"/api/download/{job_id}/{fmt}?token={job_access_token}"
    for fmt in ("docx", "html", "md")
    if fmt in report_paths
}
```

修正结果：报告生成完成后，合法用户仍可下载 DOCX/HTML/MD，未携带 token 的请求仍会被拒绝。

### 偏差 2：轮询 URL 已有查询参数时不能再简单拼接 `?t=...`

发现情况：后端返回的 `resultUrl` 已包含 `?token=...`，如果前端继续使用原逻辑 `${state.resultUrl}?t=${Date.now()}`，会形成错误查询串。

修正动作：新增 `withQuery` 函数统一合并查询参数。

```javascript
function withQuery(url, params) {
    const target = new URL(url, window.location.origin);
    for (const [key, value] of Object.entries(params)) {
        if (value !== undefined && value !== null && String(value) !== "") {
            target.searchParams.set(key, String(value));
        }
    }
    return `${target.pathname}${target.search}`;
}
```

修正结果：轮询、SSE 与下载场景都可以稳定携带 token，不会破坏已有参数。

### 偏差 3：单元测试导入被第三方 `tests` 包抢占

发现情况：第一次运行 `python -m unittest tests.test_job_access_control -v` 时，当前 Python 环境中存在第三方 `tests` 包，导致本地测试模块没有被正确导入。

修正动作：在个人项目目录的 `tests/` 下新增 `__init__.py`，明确本地测试包边界。

修正结果：再次运行单元测试，4 个授权测试全部通过。

### 偏差 4：期末报告初版只引用 Prompt 文档，未在报告正文完整体现三类记录

发现情况：初版报告虽然在 `docs/prompt-records.md` 中记录了 Prompt 和偏差，但报告正文没有完整展开“关键Prompt的完整文本、AI生成结果中与安全约束相关的关键片段、发现偏差或问题时的交互记录”。

修正动作：重写 `期末报告_苏禹宽.md`，新增第四、第五、第六章，将三类材料直接写入报告正文。

修正结果：期末报告本身即可完整呈现安全管理过程，同时仍保留 `docs/` 下的过程材料作为支撑文件。
